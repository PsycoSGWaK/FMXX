1 · Présentation du projet

FMXX est un outil web compagnon du jeu Football Manager 26. Il permet à un entraîneur de gérer, en dehors du jeu, le suivi de sa carrière : objectifs de saison par compétition, gestion d'effectif importé depuis FM, composition tactique « sub », mercato (départs et arrivées) et palmarès historique.

Fonctionnalités principales

  • Objectifs de saison — définition d'objectifs par compétition et calcul d'un taux de réussite.
  • Gestion d'effectif — import depuis FM (CSV ou HTML), filtres, tri, statuts mercato et contrats.
  • Tactique Sub — composition d'un onze, remplaçants et super-subs selon 6 formations.
  • Mercato — suivi des départs (vente / prêt / libre) et des arrivées (cible / négociation / signé) avec solde budgétaire.
  • Palmarès — historique des saisons, objectifs et résultats, avec pourcentage de réussite.
  • Back-office d'administration — gestion des utilisateurs, compétitions, clubs et pays.
  • Multilingue — interface en français, anglais et espagnol.

Public & contexte

Application personnelle hébergée localement sur WAMP (Windows / Apache / MySQL / PHP). Chaque utilisateur dispose de son propre compte et ne voit que ses données (effectif, objectifs, saisons…). Un type d'utilisateur type = '1' donne accès au back-office d'administration.

2 · Architecture & arborescence

Application multi-pages PHP « classique » (sans framework). Chaque page .php est un point d'entrée qui démarre la session, se connecte à la base via PDO, inclut l'en-tête / la barre de navigation, puis génère le HTML. Les traitements de formulaires sont isolés dans des fichiers suffixés _post.php qui valident, écrivent en base puis redirigent (motif Post/Redirect/Get).

Procédures & trigger

  • CheckUserExists(username, mail, OUT taken…) — teste l'unicité du pseudo et de l'email à l'inscription.
  • DeleteUser(idUser) — supprime un compte et toutes ses données liées (mercato, tactic, objectif, saison_meta, joueur) dans le bon ordre.
  • before_insert_competition — trigger qui empêche les doublons de compétition (même nom + type + pays) via SIGNAL SQLSTATE '45000'.

4 · Authentification & sécurité

Connexion & sessions

Les mots de passe sont stockés hachés (password_hash, bcrypt) et vérifiés avec password_verify. À la connexion réussie, la session est régénérée (session_regenerate_id(true)) et un cookie persistant auth_token (httponly, samesite Lax) est posé pour 30 jours ; auth_check.php restaure la session à partir de ce jeton tant qu'il n'est pas expiré.

// login_post.php — vérification + cookie « se souvenir de moi »
if (password_verify($_POST['password'], $resultat['password'])) {
    session_regenerate_id(true);
    $token  = bin2hex(random_bytes(64));
    setcookie("auth_token", $token, [
        'expires'  => time() + (86400 * 30),
        'httponly' => true,
        'samesite' => 'Lax',
    ]);
}

Protection CSRF

Chaque formulaire inclut un jeton via csrf_field() et chaque traitement POST sensible appelle csrf_verify(), qui compare en temps constant (hash_equals) le jeton de session.

// csrf.php
function csrf_verify(): void {
    $token = $_POST['csrf_token'] ?? '';
    if (!hash_equals(csrf_token(), $token)) {
        http_response_code(403);
        exit('Requête invalide.');
    }
}

Validation des saisies

Le module partagé validation.php centralise les règles, réutilisées à l'inscription, à la modification de profil et au reset admin :

  • Mot de passe — 14 caractères minimum, au moins une majuscule, une minuscule, un chiffre et un caractère spécial (plafond 72 octets : limite bcrypt).
  • Email — format RFC (filter_var) plus vérification de l'existence du domaine (enregistrement MX, repli A/AAAA).
  • Messages d'erreur traduits (FR/EN/ES) et indicateur de robustesse en direct côté client.
// validation.php — politique de mot de passe
function validate_password(string $pwd): ?string {
    if (strlen($pwd) > 72)                  return 'pwd_max';
    if (mb_strlen($pwd) < 14)              return 'pwd_min';
    if (!preg_match('/[A-Z]/u', $pwd))    return 'pwd_upper';
    if (!preg_match('/[0-9]/u', $pwd))    return 'pwd_digit';
    // … minuscule, caractère spécial
    return null;
}

Exposition des fichiers & secrets

  • Le dossier sql/ (et sql/api_dumps/) est protégé par un .htaccess (Require all denied) — schéma, dumps et clés ne sont pas téléchargeables.
  • La clé API-Football est stockée hors webroot (C:\wamp64\fmxx_secrets\api_key.txt, ou variable d'env FMXX_API_KEY_FILE).
  • Toutes les requêtes SQL utilisent des requêtes préparées PDO (ATTR_EMULATE_PREPARES = false).
  • Les uploads sont limités (5 Mo, extensions csv/html/htm, is_uploaded_file) et protégés CSRF.
Rappel hébergement : en production, remplacer l'utilisateur MySQL root (sans mot de passe, acceptable en local) par un compte dédié à privilèges restreints dans db.php.

5 · Parcours & pages

Un visiteur non connecté voit la page d'accueil (présentation + modals connexion/inscription). Une fois connecté, index.php devient le tableau de bord, hub vers les autres pages. Le diagramme de navigation complet figure en section 9.

index.php — Tableau de bord
Onglets : Objectifs de la saison (par compétition, % de réussite), Budget (transferts / salaires), Effectif (liste, filtres, import, vidage), Tactique Sub (onze + remplaçants selon la formation). Inclut le modal d'import d'effectif.
mercato.php — Mercato
Départs répartis en vente / prêt / libre (recettes calculées sur prixDemande) et arrivées en cible / négociation / signé (dépenses), avec calcul du solde = recettes − dépenses.
joueur.php — Fiche joueur
Détail et édition d'un joueur (identité, poste, contrat, statut mercato). Vérifie systématiquement que le joueur appartient à l'utilisateur connecté avant toute modification.
palmares.php — Palmarès
Historique par saison (saison_meta) des objectifs et résultats, avec calcul du pourcentage de réussite à partir d'un barème de classements.
admin.php — Administration (type = '1')
Back-office à onglets : gestion des utilisateurs (rôle, reset mot de passe, suppression), compétitions, clubs et pays (ajout / édition / suppression).
Modals (navbar) — Paramètres · Profil · Suppression de compte
Paramètres (pays, genre, division, club, saison, coupe d'Europe), profil (pseudo, email, mot de passe), suppression de compte. saison_next.php fait passer à la saison +1.

6 · Import de données

Import de l'effectif (depuis FM)

upload.php accepte un export FM au format CSV (séparateur ;) ou HTML (table). Les en-têtes sont normalisés puis mappés vers les colonnes de la table joueur via une table de correspondance COL_MAP qui gère les libellés FM24/FM26 en français comme en anglais. L'ancien effectif est remplacé.

// upload.php — correspondance en-têtes FM → colonnes BDD (extrait)
const COL_MAP = [
    'joueur' => 'nom',   'name'   => 'nom',
    'âge'    => 'age',   'buts'   => 'buts',  'goals' => 'buts',
    'prix demandé' => 'prixDemande', 'asking price' => 'prixDemande',
    // …
];

Import des équipes (API-Football)

Chaîne en 2 scripts CLI (jamais exécutés via le web), pour peupler la table equipe à partir de l'API-Football (saison 2024, plan gratuit) :

  1. fetch_teams.php — récupère un dump /teams par ligue dans api_dumps/<ligue>.json (idempotent, pause anti rate-limit, arrêt propre sur quota).
  2. import_teams_from_dumps.php — lit les dumps et génère equipe_insert.sql (dédoublonné), à injecter en base.
// fetch_teams.php — clé lue HORS webroot
$keyFile = getenv('FMXX_API_KEY_FILE') ?: 'C:/wamp64/fmxx_secrets/api_key.txt';
$ch = curl_init("https://$host/teams?league=$leagueId&season=2024");
curl_setopt_array($ch, [
    CURLOPT_HTTPHEADER => ["x-apisports-key: $apiKey"],
]);
Le détail du pipeline (fichiers, ligues, états) est schématisé en section 9.

7 · Internationalisation

L'interface est disponible en français, anglais et espagnol. lang.php lit la langue dans $_SESSION['lang'] (par défaut fr) et charge le tableau $t depuis lang/<langue>.php. Les pages affichent les libellés via $t['clé'] ; lang_post.php change la langue active.

// lang.php
$lang = $_SESSION['lang'] ?? 'fr';
if (!in_array($lang, ['fr', 'en', 'es'])) $lang = 'fr';
require_once __DIR__ . "/lang/{$lang}.php";   // définit $t = [ … ]

Les messages de validation (mot de passe, email) sont eux aussi traduits : les fonctions renvoient un code stable, traduit à l'affichage par validation_message($code, $t).

8 · Installation & déploiement

  1. Prérequis — WAMP (Apache + PHP 8.x + MySQL/MariaDB). Placer le projet dans www/FMXX.
  2. Base de données — créer la base fmxx, importer le schéma et les procédures / trigger, puis (optionnel) equipe_insert.sql pour les clubs.
  3. Connexion — vérifier les identifiants dans db.php (local : root sans mot de passe).
  4. Clé API (optionnel) — créer C:\wamp64\fmxx_secrets\api_key.txt avec la clé API-Football, puis lancer en CLI php sql/fetch_teams.php et php sql/import_teams_from_dumps.php.
  5. Accès — ouvrir http://localhost/FMXX/ et créer un compte.
Avant mise en ligne : créer un utilisateur MySQL dédié (mot de passe fort, privilèges limités), activer HTTPS (le cookie d'auth passe alors en secure), et vérifier que les .htaccess sont bien pris en compte par l'hébergeur.

9 · Schémas & diagrammes

9.1 · Modèle conceptuel (MCD)

Vue conceptuelle (méthode Merise) : les entités (rectangles) reliées par des associations (ovales, avec un verbe) et leurs cardinalités. À ce niveau il n'y a pas encore de clés étrangères — elles apparaissent dans le modèle relationnel (9.2).

Lecture Merise : « un UTILISATEUR possède 0,n JOUEUR ; un JOUEUR appartient à 1,1 UTILISATEUR ». Les référentiels PAYS / ÉQUIPE / COMPÉTITION sont partagés entre utilisateurs.

9.2 · Modèle logique / relationnel (MLD)

Relations logiques (par idUser, idPays, idCompetition, idEquipe). La flèche pointe vers la table référencée.

9.3 · Architecture technique (flux d'une requête)

Le code PHP s'exécute côté serveur : seul le HTML/CSS/JS atteint le navigateur. L'import API-Football se fait en ligne de commande, séparément.

9.4 · Séquence d'authentification

Mot de passe vérifié → régénération de session → cookie httponly. Les visites ultérieures restaurent la session via le jeton.

9.5 · Pipeline d'import

Deux chaînes distinctes : l'effectif passe par le web (upload), les équipes par des scripts en ligne de commande.

9.6 · Plan de navigation

Hub central (tableau de bord) reliant les pages. Les fenêtres modales (paramètres, profil…) sont disponibles depuis la barre de navigation.
